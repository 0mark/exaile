  1  # Provides a signals-like system for sending and listening for 'events' 
  2  # 
  3  # 
  4  # Events are kind of like signals, except they may be listened for on a  
  5  # global scale, rather than connected on a per-object basis like signals  
  6  # are. This means that ANY object can emit ANY event, and these events may  
  7  # be listened for by ANY object. Events may be emitted either syncronously  
  8  # or asyncronously, the default is asyncronous. 
  9  # 
 10  # The events module also provides an idle_add() function similar to that of 
 11  # gobject's. However this should not be used for long-running tasks as they 
 12  # may block other events queued via idle_add(). 
 13  # 
 14  # Events should be emitted AFTER the given event has taken place. Often the 
 15  # most appropriate spot is immediately before a return statement.
